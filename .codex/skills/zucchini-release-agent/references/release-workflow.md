# Release Workflow

## Scope

This skill is for the Zucchini repository only.

The release pipeline is defined in [`.github/workflows/release.yml`](../../../.github/workflows/release.yml). Pushing a tag that matches `v*` triggers the workflow.

## Current Repo Rules

- The version source of truth is [`apps/desktop/package.json`](../../../apps/desktop/package.json).
- Release notes and `CHANGELOG.md` are generated by [`apps/desktop/scripts/generate-changelog.mjs`](../../../apps/desktop/scripts/generate-changelog.mjs).
- The release commit format is `chore: release <version>`.
- The tag format is `v<version>`.
- GitHub Actions creates or updates a draft release after the build jobs pass.
- Artifact publishing happens in the release workflow after the draft exists.

## Versioning Rules Used By The Script

- `--channel prerelease`
  - If the current desktop version is `X.Y.Z-beta.N`, increment it to `X.Y.Z-beta.(N+1)`.
  - If the current desktop version is stable `X.Y.Z`, default to `X.Y.(Z+1)-beta.1`.
- `--channel main`
  - If the current desktop version is `X.Y.Z-beta.N`, publish `X.Y.Z`.
  - If the current desktop version is already stable, keep `X.Y.Z` only when tag `vX.Y.Z` does not already exist.
  - If tag `vX.Y.Z` already exists, require an explicit `--version`.

## Preconditions

- The git working tree must be clean.
- The current branch must be `main`.
- Local `HEAD` must equal `origin/main`.
- `gh` must be installed and authenticated.

## Publication Rules

- Beta releases are published as GitHub prereleases.
- Stable releases are published as the latest GitHub release.
- The release title is `Zucchini <version>`.

## Failure Handling

- If the workflow run never appears, fail instead of guessing.
- If the workflow run fails, surface the failed logs and stop.
- Do not delete tags or rewrite history automatically.
